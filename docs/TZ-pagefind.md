# ТЗ: Поиск (Pagefind) + облегчённый индекс + единый путь по авторам

> Статус: план (обсуждён 2026-07-09). Реализация — в Треке 1 Фаза D.
> Цель: полнотекстовый поиск по всему сайту без раздувания индекса + унифицировать поиск авторов.

## 1. Проблема
`js/search.js` ищет только по `title/oneliner/description` в `articles-index*.json`. Полный текст статей
не индексируется. Индекс тяжёлый: 3 файла на язык × ~55 КБ на 23 статьи; на масштабе — десятки МБ на язык.
Самое тяжёлое в записи — `description` + `threads` (полнотекст). Авторы ищутся отдельным путём (`@author`).

## 2. Решение
| Компонент | Роль |
|---|---|
| **Pagefind** | Полнотекстовый поиск по ВСЕМУ HTML (статьи ×3 версии, теги, законы, учёные, **авторы**) — чанки 2–10 КБ, браузер грузит только нужный (~3 КБ/слово); подсветка/сниппеты/фильтры из коробки |
| **облегчённый `articles-index*.json`** | Только мета для ЛЕНТЫ и структурных фильтров: `id, version, title, oneliner, tags, scientists, authors, url, date, thumbs, reading, categories`. **Дроп `description` + `threads`** (их даёт Pagefind) |

### Единый путь по авторам
- Страницы авторов — HTML → Pagefind их индексирует → **имя автора ищется обычным полнотекстом**, как теги/учёные/статьи (без спец-синтаксиса) = «одним путём».
- `@author` / `#tag` / `!scientist` ОСТАЮТСЯ как структурные фильтры ленты поверх облегчённого индекса.
- Итог: Pagefind = свободный поиск по всему; `@ # !` = точная фильтрация.

## 3. Что убираем / оставляем в `search.js`
- ❌ `description`, `threads` из индекса; ❌ полнотекст-поиск по индексу (его берёт Pagefind).
- ✅ Фильтры `#tag` / `@author` / `!scientist`; ✅ выпадающие подсказки; ✅ лента последних (из облегчённого индекса); ✅ календарь/категории.

## 4. Размер индекса (экономия ~70%)
| Статей | Было | Стало |
|---|---|---|
| 5 000 | 2.5 МБ | 0.75 МБ |
| 100 000 | 50 МБ | 15 МБ |

## 5. Интеграция
```bash
python run.py html          # генерация HTML
npx pagefind --site lang    # построение индекса (в конце cmd_html)
```
- Шаблоны: `data-pagefind-body` на основной контейнер; `data-pagefind-filter="lang|version|type"`; веса `data-pagefind-weight` на h1/oneliner; `data-pagefind-ignore` на служебное (архив/about/навигация).
- UI: `pagefind-ui.css` + `pagefind-ui.js` + `<div id="search">`; на главной — лента + поиск (появляется при фокусе).
- ⚠️ **`npx`/Node в текущем окружении НЕТ** — Pagefind строится на билд-машине/CI. Локально здесь не прогнать: интеграцию делаем как шаг сборки (не в этом окружении) либо ставим Node.

## 6. Связь с другими задачами
- **Главная (Трек 1 Фаза D):** Pagefind + два поля-фильтра (по тегам / по законам).
- **Индексы-шардинг (`docs/TZ-indexes.md`):** Pagefind снимает остроту (полнотекст ушёл из индекса); шардинг ленты по месяцам — только если сама ЛЕНТА станет тяжёлой на десятках тысяч.
- **Дубль авторов:** поле `authors` в индексе + `authors-graph.json` — оставляем (graph нужен для облака/страниц/соавторов); Pagefind поверх, не вместо.

## 7. Открытые вопросы
- Дедуп в выдаче: одна статья = 3 версии (3 HTML) → фильтр Pagefind по `version` + предпочесть popular.
- RTL/арабский — проверить токенизацию Pagefind.
- Не коммитить бинарный индекс Pagefind? Строить на деплое.
