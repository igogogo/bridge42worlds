# ТЗ: Реструктуризация индексов (шардинг) — bridge42worlds

> Статус: план (не реализовано). Задача #33.
> Цель: убрать блокер масштаба — сейчас лента грузит весь индекс в браузер.
> Актуально на 2026-07-08.

## 1. Проблема
Клиент (`js/search.js`) при открытии ленты грузит **весь** `articles-index*.json`
целиком (3 файла: popular/simple/advanced для текущего языка). На 10 000 статей
это **10–30 МБ на язык на каждую загрузку страницы** → сайт становится неюзабельным
уже на ~1–2K статей. Граф (800 узлов) — НЕ проблема; блокер именно индекс статей.

## 2. Что сейчас (as-is)
- Файлы: `lang/{lang}/articles-index.json` (popular), `articles-index-simple.json`, `articles-index-advanced.json`.
- Запись: `update_index()` (живой путь, `generate.py` ~389) и `_index_entry()` +
  `rebuild_indexes()` (пересборка, `generate.py` ~1490). **Оба должны писать одинаковые поля** (правило дрейфа!).
- Запись в индексе (текущие поля): `id, version, title, oneliner, description, threads, thumbs, authors[:50], date, tags, scientists, url, reading, categories, primary_category`.
- Чтение (`search.js`): грузит 3 версии-файла в `searchIndex`, фильтрует по `item.version === effVersion()`.
  Используется: лента (`showLatest`), поиск (`doSearch`), страницы тег/учёный/автор (`applyPageContext`),
  «похожие»/«следующая» (`js/scroll.js` — грузит `articlesIndex`), случайная статья (нет пока), фильтры по категории/дате.

## 3. Что нужно (to-be)
Разложить монолит в папку **`lang/{lang}/index/`**, каждая страница грузит только нужное (КБ, не МБ).

### 3.1 Структура папки
```
lang/{lang}/index/
  manifest.json              # {lang, updated, months:["2026-07",...], count, versions:["popular","simple","advanced"]}
  ids.json                   # [["2607.00433v1","2026-07-01"], ...]  — id→date для всех статей (случайная/навигация)
  latest.json                # свежие ~150 записей (popular) для ленты по умолчанию
  months/2026-07.json        # шард месяца: {popular:[...], simple:[...], advanced:[...]} ИЛИ по версии-файлу
  tags/{tag}.json            # [refs] статей с этим тегом (пер-сущностный список)
  authors/{slug}.json        # [refs] статей автора
  scientists/{slug}.json     # [refs] статей учёного
  laws/{law}.json            # [refs] статей, где встречается закон (через теги)
```
- **ref** (облегчённая запись для списков) = `{id, title, date, url, oneliner}` (без threads/50-авторов/description — их держим ТОЛЬКО в data.json/полном шарде).
- Полная запись (с description/threads/thumbs/authors) остаётся в **месячных шардах** и в `data.json`.
- Лента (карточка) требует description/thumbs/authors → грузит **latest.json** (полные записи) + догрузка месячных шардов на скролл.
- Пер-сущностные списки (страница тега и т.п.) — облегчённые ref (title+date+url+oneliner достаточно для списка).

### 3.2 Форматы (пример)
`manifest.json`:
```json
{"lang":"ru","updated":"2026-07-08","versions":["popular","simple","advanced"],
 "months":["2026-07"],"count":10}
```
`ids.json`:
```json
[["2607.00433v1","2026-07-01"],["2607.00982v1","2026-07-01"]]
```
`latest.json` — массив ПОЛНЫХ записей (как сейчас в articles-index.json, popular), но только свежие N.
`months/2026-07.json`:
```json
{"popular":[<полные записи>],"simple":[...],"advanced":[...]}
```
`tags/black_hole.json`:
```json
[{"id":"2607.00433v1","title":"...","date":"2026-07-01","url":"/lang/ru/archive/.../index.html","oneliner":"..."}]
```

## 4. Изменения в генерации (`generate.py`)
- **Новая функция** `write_index_shards(lang)` (вызвать из `rebuild_indexes()` и в конце `process_day`):
  1. Собрать все записи из `data.json` (через `iter_articles()`), по версиям.
  2. Записать `index/months/{YYYY-MM}.json` (сгруппировать по `date[:7]`).
  3. Записать `index/latest.json` = последние N popular (сортировка по date desc, N=150).
  4. Записать `index/ids.json` и `index/manifest.json`.
  5. Пер-сущностные: пройти все статьи, разложить refs по тегам/авторам/учёным/законам → `index/{kind}/{key}.json`.
- **Совместимость:** на переходный период можно оставить старые `articles-index*.json` (или редирект-обёртку), пока клиент не переключён. Лучше — фича-флаг `config.sharded_index`.
- **Правило дрейфа сохраняется:** поля ref/полной записи определить в ОДНОМ месте (хелпер `_ref(data)` / `_full_entry(...)`), звать из обоих путей.

## 5. Изменения в клиенте (`js/search.js`, `js/scroll.js`)
- **Лента:** грузить `index/latest.json` вместо трёх полных файлов. «Показать ещё»/скролл → догрузка `index/months/{месяц}.json` по убыванию месяцев (из manifest).
- **Страницы тег/учёный/автор:** вместо фильтра всего индекса — грузить `index/{kind}/{key}.json` (маленький). `applyPageContext` больше не нужен для этих страниц.
- **Случайная статья:** грузить `index/ids.json`, `const [id,date]=ids[rand]; location.href = archive/{date}/{id}/index.html`.
- **«Похожие»/«следующая»:** сейчас берут весь индекс. Варианты: (а) считать «похожие» на билде и класть в `data.json.related` (список id) → клиент просто рендерит; (б) грузить только latest+текущий месяц. Рекомендую (а) — детерминированно и дёшево.
- **Поиск:** **Pagefind** (`npx pagefind --site .`) — статический поисковый индекс, грузится порциями. Отдельная интеграция (Фаза 3). До неё — поиск ограничить latest+подгруженными шардами.

## 6. Фазы (чтобы не ломать всё разом)
- **Фаза 1 (безопасная, без Pagefind):** сгенерить папку `index/` (manifest+ids+latest+months+пер-сущностные) ПАРАЛЛЕЛЬНО со старыми файлами. Переключить клиент за фича-флагом. Проверить лента/тег/автор/случайная.
- **Фаза 2:** ленивая догрузка месяцев на скролл; `related` считать на билде.
- **Фаза 3:** Pagefind для поиска; удалить старые монолит-индексы.

## 7. Критерии приёмки
- Открытие ленты грузит ≤ ~300 КБ (latest), не весь индекс.
- Страница тега грузит только свой `tags/{tag}.json`.
- Случайная статья работает без полного индекса.
- «Похожие», фильтры категорий/дат, переключатель версий и «мини» — работают.
- `run.py html` / `reindex` собирают папку `index/`; `run.py check` чист.
- Кэш-версии ассетов подняты (`?v=N`).

## 8. Нюансы
- **RTL (ar):** ничего особого, те же файлы на язык.
- **Мини-режим** (`threads`) — только на главной, берёт latest (там threads есть в полной записи).
- **Не тащить** description/threads/50-авторов в пер-сущностные ref — только в latest/месячные/ data.json.
- Держать `iter_articles()` как единственный источник правды (data.json).
