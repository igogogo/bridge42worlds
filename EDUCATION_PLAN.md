# Раздел «Education» — план (черновик 2026-07-14)

Цель: не 1 статья на каждый из 94 законов, а ~50 связных образовательных статей,
каждая объединяет несколько близких по смыслу законов/принципов/теорем/уравнений/эффектов
из `laws.json` в один нарратив (как «Три закона Ньютона» — уже сделано,
`reports_theory/newton_laws_article.html`, референс-стиль — `reports_theory/vertical_wheel_article.html`).

## Принцип группировки
Группируем по физическому/математическому СМЫСЛУ, не по алфавиту/типу. Внутри группы —
один сквозной пример/вывод, не набор изолированных абзацев. У каждой статьи — секция
«углубление» (принцип наименьшего действия / философия / связь с другими законами, как §6
у Ньютона) и раскрывающиеся задачи с чистыми числами (Дано→Закон→Решение→Ответ).

## DRY с существующими карточками законов
- НЕ дублируем историю открытия — она уже есть на `/lang/{lang}/laws/{id}.html` (advanced-уровень).
  Образовательная статья даёт ссылку + короткую пометку значимости, не переписывает историю.
- Значимость/суть можно тянуть прямо из `laws.json[id]["description"]["popular"|"simple"]`
  как затравку для вступления к под-разделу — не сочинять с нуля.
- Упоминания тегов — сразу как `class="text-tag" data-tag="..."` (готовый hover-тултип из
  search.js, ничего изобретать не надо, просто использовать существующий паттерн).
- Граф знаний (tags⇄laws⇄scientists) — на страницу закона можно добавить обратную ссылку
  «📘 разобрано подробнее в: {education-статья}», симметрично тому как теги ссылаются на законы.

## РАСШИРЕННЫЙ ПРОМТ — как делать статью «Education» (накопленный опыт, 2026-07-14)

**Режим работы: НЕ автомат.** Юзер явно: «мы с тобой здесь генерим, вычитываем, проверяем — это
полуручная ювелирная работа». Каждая статья — цикл генерация→показ→правки по фидбеку, не «сделал
и забыл». НЕ штамповать 52 статьи молча одним проходом — делать партиями, показывать, вносить правки
по замечаниям (как с Ньютоном: 4 раунда правок — анимации/единицы/§6, потом «точки слишком
однотипные», потом «числа для задач», потом тайминг+живые числа+фидбек-блок).

### 1. Структура документа (см. `reports_theory/newton_laws_article.html` как эталон)
- Самодостаточный HTML: инлайн `<style>`, CSS-переменные (paper/ink/brass/steel/moss), тема
  light/dark из коробки (`@media (prefers-color-scheme)` + `:root[data-theme]`).
- Sticky TOC слева (`.toc`, виден от 1020px), hero с h1+подзаголовком+анимированной диаграммой,
  `.abstract` (краткая аннотация), секции `§N` с `.sec-head`+`hr.rule`+`.measure` (ширина 680px
  для текста, формулы/иллюстрации могут быть шире).
- Формулы — HTML/CSS (span.vec для векторов, sub/sup), НЕ KaTeX/LaTeX — визуально спокойнее для
  «рабочей заметки», плюс `.eq`(нумерованная) и `.plate`(врезка с итоговой формулой + `.cap`).
- `.callout` (brass/steel/moss) — параллели с бытовыми примерами/частыми ошибками.
- Секция «углубление» (философия+матаппарат+связь с другими законами через принцип наименьшего
  действия/теорему Нётер/итд, как §6 у Ньютона) — обязательна для каждой статьи, это то, что
  отличает «образовательную статью» от простого пересказа. Шрифт в этой секции чуть мельче
  (font-size .94rem против базовых 18px) — визуально «плотнее», как принято для advanced-глубины.
- Задачи в конце (`<details class="problem">`) — закрыты по умолчанию («сначала подумай сам»),
  структура ответа ВСЕГДА: **Дано → Закон → Решение → Ответ** (`.qlabel` для меток).
  Числа ВСЕГДА подбираются так, чтобы ответ выходил целым/чистым (не 3.33, не 1.1667) — это
  стандартный методический приём, не опция. Где физически осмысленно — g≈10 м/с² вместо 9.8 с
  явной оговоркой-сноской «почему округлили» (`.units`), чтобы считалось в уме.
- **Шрифт блока задач мельче основного текста** (юзер-фидбек 2026-07-14): `.problem` целиком —
  `font-size:.9rem` (было унаследовано от body 18px/1rem, визуально «весило» так же, как обычный
  текст — задачи должны читаться заметно компактнее, как факультативный материал). `summary` —
  `.9rem` явно, `.answer` — `.86rem` (немного мельче самого summary, консистентно с уже принятым
  паттерном «advanced/углубление мельче основного» из §6-правила выше).

### 2. Анимации — ГЛАВНОЕ отличие от статичной картинки
- Принцип: «всё, что можно анимировать — анимируй». НЕ абстрактные одинаковые точки — РЕАЛЬНЫЕ
  объекты (груз/шайба/шарик/ракета), размер визуально кодирует величину (масса и т.п.), сила —
  видимая стрелка, физически прикреплённая к объекту (едут вместе одной SVG-группой/`<g>`, а
  не порознь) — см. hero-диаграмму Ньютона (F+груз едут вместе, толкает сзади).
- **Тайминг (важно, юзер поправил отдельно):** один цикл — НЕ быстрее ~5 секунд суммарно.
  Если процесс сам по себе быстрый физически — не гнаться за реализмом скорости, брать медленнее,
  чтобы глаз успевал следить. После того как объект доехал/событие завершилось — ДЕРЖАТЬ финальное
  положение 1-2 секунды («дать подышать, переварить») ПЕРЕД тем как цикл начнётся заново — не
  дёргать сразу на рестарт. В CSS `@keyframes` это делается так: движение занимает первые ~60-70%
  временной шкалы, затем `100%{ /* та же конечная позиция */ }` — держит паузу перед `infinite`-
  рестартом. Если сравниваешь несколько объектов с разной скоростью в одной сцене (как 3 груза
  §3 у Ньютона) — используй ОДИНАКОВУЮ полную длительность анимации на все объекты (иначе они
  разъедутся по фазе при зацикливании) и разный % на шкале, где каждый доезжает до финиша — тогда
  каждый цикл все стартуют «с одной линии старта» синхронно.
- **Функциональная (математическая) анимация — новое, добавлять впредь:** мало просто анимировать
  объект в схематичной сцене — рядом должны динамически меняться ЧИСЛА (F, a, v, x/t — что
  уместно для конкретного закона) синхронно с движением, и/или маркер-точка должна двигаться ПО
  ГРАФИКУ функции (v(t), a(t) и т.п.), а не только по схеме сцены — это и называется юзером
  «функциональная анимация»: связка объект↔число↔график в реальном времени, как в физическом
  симуляторе, а не три разрозненных статичных элемента (картинка+число+PNG-график). Технически
  это требует чуть-чуть JS (`requestAnimationFrame`, считать значения по известной формуле —
  не читать computed CSS, а вычислять напрямую, чтобы числа были точные) — использование
  небольшого инлайн `<script>` для этого ДОПУСТИМО и ожидаемо (референсный документ
  `vertical_wheel_article.html` тоже использует JS для интерактивных виджетов).

### 3. Обратная связь — обязательный блок
Внизу каждой статьи — явный призыв сообщать об ошибках: «Нашли неточность? Напишите нам — мы
всё учтём, разберём и внесём правки» + контакт (email или форма — см. как на статьях сайта уже
есть `build_feedback_html()`/«Отклик», можно переиспользовать паттерн). Формулировка должна явно
обещать, что фидбек РЕАЛЬНО обрабатывается, не «в стол».

### 3б. «Эксперимент дома» — обязательный раздел (юзер-фидбек 2026-07-14)
Отдельная секция ПЕРЕД задачами (или сразу после «углубления») — 1-3 предложения эксперимента,
который можно реально сделать дома простыми подручными предметами, с вау-эффектом (не скучное
«измерьте линейкой», а что-то наглядно удивляющее). Примеры уровня качества: вращающийся стул +
гантели/книги в руках (сохранение момента импульса — руки к телу, скорость вращения растёт на
глазах); ньютоновские шары/маятник из подшипников на нитках; фокус со скатертью (уже есть у
Ньютона §2, канонический пример именно этого рода); тарелка с водой на вращающемся стуле (гироскоп-
эффект). Формат — `.callout` с новым тоном `experiment` (переиспользовать `.callout` палитру, тег
«Эксперимент дома», можно смайл 🧪 в подписи), 2-4 пункта: что взять, что сделать, что заметить
(«почему это удивительно» — явно связать наблюдение с законом статьи, не оставлять как трюк без
объяснения).
